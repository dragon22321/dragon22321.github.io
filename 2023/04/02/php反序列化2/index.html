phar反序列化

我们时常会看到以.jar作为后缀的文件，这个其实java归档的一种打包方式，同样的，phar也是php归档的一种打包方式，而之所以能作为反序列化的利用点，就是因为phar文件本质上是一种压缩文件，会以序列化的形式存储用户自定义的meta-data。当受影响的文件操作函数调用phar文件时，会自动反序列化meta-data内的内容。

那么什么是mete-data呢？

这里我们先来了解一下，phar一个包的组成

stub:phar文件的标志，必须以 , 结尾，否则无法识别。xxx可以为自定义内容。

stub这一部分，在了解时发现其实这样写 < __HALT_COMPILER();> 也是可以正常解析的

manifest:phar文件本质上是一种压缩文件，其中每个被压缩文件的权限、属性等信息都放在这部分。这部分还会以序列化的形式存储用户自定义的meta-data，这是漏洞利用最核心的地方。

content:被压缩文件的内容

signature (可空):签名，放在末尾。

接下来就是我们常见的构造一个phar文件的过程

class a{

public $b

}

$c=new a;

$phar=Phar(a.phar)//这步是为了利用Phar这个类来构造我们的phar文件，（）里面的内容决定了我们的文件名，后缀一定是phar。

$phar->startBuffering()//调用这个函数，达到一个初始化的目的，写入之前开始缓冲Phar写入操作

$phar->setStub(““);//设置stub，因为之前我们说了，末尾一定要是才能被识别为phar文件

$phar->setMetadata($p);//将自定义的meta-data存入manifest，这一部分就是把序列化的对象加入其中，为我们之后的反序列化作铺垫。

$phar->addFromString(“test.txt”, “test”);//添加要压缩的文件（这一步必须要有，如果不设置就无法生成phar文件，文件可以是任意文件）

$phar->stopBuffering();这是最后的签名

当我们执行完这串代码之后，我们的phar文件会自动地生成在该文件的目录下

但是想要利用这个文件，少不了一些特定的函数，这里从csdn隐形卟师傅这里拿到了总结的一些函数，存在反序列化的漏洞

而这里，我们需要使用到我们之前学过的一个伪协议，phar://协议，这个协议能够解压文件，php大部分的文件系统函数在通过phar://伪协议解析phar文件时，都会将meta-data进行反序列化，在这里就被利用到反序列化中，具体例子还是看师傅的例子

test) ; } } is_dir('phar://test.phar');//以phar伪协议的格式传入 ?>

当然，如果遇到phar://不能出现在开头的过滤条件，那么可以用下面的方式绕过

compress.bzip://phar:///test.phar/test.txt

compress.bzip2://phar:///test.phar/test.txt

compress.zlib://phar:///home/sx/test.phar/test.txt

php://filter/resource=phar:///test.phar/test.txt

这个反序列化的漏洞可以利用在特定格式的文件上传当中，比如需要上传一个gif文件，那么文件头就可以进行修改

$phar->setStub(“GIF89a“);但是我们需要找到一个带有魔术方法的类，除此之外，phar伪协议还能被正常利用，这样才能发挥最大作用。

php-session反序列化

关于session在之前的csrf学习中已经有所了解，所以就不做过多的赘述，简单来说，当客户端访问服务器端时，如果没有session-id，那么服务端就会返回一个session-id给客户端，只要浏览器不关闭，就会一直在，而这里，我们就要从服务端的角度来看一下生成session-id的原理。

1、这里我们说一下php中的Session机制

说到这个，肯定离不开 session_start()函数，它的作用是打开Session，并且随机生成一个32位的session_id，session的全部机制也是基于这个session_id，服务器就是通过这个唯一的session_id来区分出这是哪个用户访问的。而session_id的值存储在服务器的临时目录tmp/temp下，存储文件格式为：sess_ + session_id，因此，简单来说，当我们在浏览器中，HTTP请求一个页面时，其中后端php执行session_start()会打开一个会话，此时会寻找COOKIE中的PHPSESSID 是否为空，若不为空session_id()就是这个值，若为空则随机生成一个32位session_id存入cookie中的PHPSESSID中，然后在对应临时文件路径下生成一个文件名为sess_+session_id 的文件以及将$_SESSION的键值对按照一定格式存储到文件中。这里就要说到利用$SESSION[]进行一个赋值，如果不进行赋值的话，那么sess + session_id这个产生的文件内部就不会有任何内容，而当我们进行了一个赋值，如$_SESSION[x]=sdsd，那么对应的，文件内部会有与之相匹配格式的序列化字符串。

而在session的产生机制有了一定的了解之后，我们来看一下php中关于session的一些相关配置，

session.auto_start//如果这个配置的选项为on，那么我们就不需要引进session_start()函数，服务器端会自动生成一个session-id

session.save_path//决定sess_session_id的保存路径

session.save_handler//用户自定义session存储的选项，默认是files 即文件形式存储

session.serialize_handler//里面是序列化存储格式

重点就在于这个session.serialize_handler，这也是php-session反序列化漏洞的原因所在。

2、php存储session的格式

这里，我们要知道php存储session的三种格式php_serialize, php_binary，php，

首先需要利用ini_set(‘session.serialize_handler’, ‘php’);来设置session的存储格式，

php_serialize:将数据以序列化的格式存储

php：数组中的键和值中间用 | 来分割，值如果是数组或对象按照序列化的格式存储,也是常用的默认格式

php_binary：处理器的格式是键名的长度对应的 ASCII 字符 ＋ 键名 ＋ 经过 serialize() 函数序列化处理后的值；注意这个键名的长度所所对应的ASCII字符，就比如说键名长度为4，那它对应的就是ASCII码为4的字符，是个不可见字符EOT
那么知道了这些，我们就来看看漏洞原理，

用session.serialize_handler = php_serialize存储的字符可以引入 | , 再用session.serialize_handler = php格式取出$_SESSION的值时 “|”会被当成键值对的分隔符，意思就是说，当我们以session.serialize_handler = php_serialize格式传入一个带有 “|”的序列化字符串时，这个字符串会以正常序列化的字符串存在，而如果我们再调用session.serialize_handler = php格式，此时就会反序列化 “|”之后的对象。当然，这里我们需要session_start()函数。

session_start()函数，这个函数就是我们不需要unserialize（）函数但是也可以造成反序列化的原因session_start重用现有会话时，php内部调用会话管理器的read函数，通过read回调函数调用现有会话数据(文件里序列化内容)，php会自动反序列化数据 填充到$_SESSION 数组中，也就是说，如果我们传入一个序列化的数据，那么在我们刷新页面之后，由于session_id和session_start()这个函数还在，那么，就会对里面的内容进行一个反序列化，从而导致漏洞的产生。
